## 架构

### ARCH-001: 新增模块不改现有文件
- **日期**: 2026-02-28
- **问题**: 给感知层新增 unit-registry / aligner / scorer 三个模块时，需要决定是否修改 perception-pipeline.ts 来集成
- **决策**: 遵循"新增不修改"原则，三个模块独立存在，通过各自的单例工厂函数对外暴露
- **理由**: perception-pipeline.ts 已在生产路径上稳定运行，修改它引入回归风险。新模块可通过上层编排器（如预处理管线）独立调用
- **预防**: 新增模块时优先检查是否能独立运行，而非强行塞入已有管线

### ARCH-002: 单例 + get/reset 工厂模式
- **日期**: 2026-02-28
- **问题**: 基础设施服务如何被多个调用方共享且可测试
- **方案**: 导出 `getXxx()` 获取单例 + `resetXxx()` 清理单例（测试用）
- **已有实例**: conditionNormalizerEngine, anomalyEngine, UnitRegistry, DataQualityScorer

## 数据

### DATA-001: DWG 文件无法直接用 ezdxf 读取
- **日期**: 2026-02-28
- **问题**: ezdxf 仅支持 DXF 格式，直接读取 DWG 文件会静默失败
- **解决**: 双轨策略 — 主路径从文件名/目录结构提取编码和关系，增强路径尝试 ezdxf（需先用 ODA File Converter 转换）
- **预防**: 涉及工业文件格式时，先确认开源库的实际支持范围，不要相信文档的"支持 DWG"声明

### DATA-002: 长缺口不应插值填补
- **日期**: 2026-02-28
- **问题**: 跨设备时间对齐时，长时间数据缺口如何处理
- **决策**: 短缺口（<3x 采样周期）线性插值；长缺口（>10x）标记为低质量并保持最后值
- **理由**: 长缺口插值会引入虚假数据，下游算法（如 FFT）会产生伪频率成分，物理约束优先原则要求不造假数据

## 调试

### DEBUG-001: 端口 3000 被占用时自动切换
- **日期**: 2026-02-28
- **现象**: `pnpm dev` 启动时端口 3000 被占用，服务自动切换到 3002
- **原因**: 其他进程（或上一次未完全退出的 dev server）占用了 3000
- **处理**: 服务内置端口回退逻辑，无需手动干预；如需固定端口可 `lsof -i :3000` 排查

## 有效指令模式

### PATTERN-001: 先列文件清单，再批量实现
- **场景**: 需要新增多个相关模块时
- **模式**: 先让 AI 列出所有相关现有文件（路径+行数+状态），理解全貌后再动手
- **效果**: 避免重复实现已有功能，确保接口兼容

### PATTERN-002: 输出文件行数作为验收标准
- **场景**: 要求新建多个文件时
- **模式**: 实现完成后用 `wc -l` 确认行数，作为基本的"非空/非 stub"验收
- **效果**: 快速验证输出物不是空壳

### PATTERN-003: pnpm dev 作为编译验证
- **场景**: 新增 TypeScript 文件后
- **模式**: 运行 `pnpm dev` 确认无编译错误
- **效果**: 比 `pnpm check` 更快的反馈循环（Vite 增量编译）
