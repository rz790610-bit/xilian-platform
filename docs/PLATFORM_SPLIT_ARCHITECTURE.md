│ POST /api/alerts/{alertId}/acknowledge                                  │
│ POST /api/alerts/{alertId}/resolve                                      │

│ 应用平台 → 赋能平台的反馈通道（用于进化飞轮）                               │

### 2.2 数据隔离原则

| 数据类型 | 赋能平台 | 应用平台 | 说明 |
|---------|---------|---------|------|
| 原始传感器数据 | ✅ 完整访问 | ❌ 不可见 | 仅赋能平台处理 |
| 特征工程结果 | ✅ 完整访问 | ❌ 不可见 | 算法细节保密 |
| 模型参数/权重 | ✅ 完整访问 | ❌ 不可见 | 核心 IP 保护 |
| 物理机理公式 | ✅ 完整访问 | ❌ 不可见 | 港机制造商知识 |
| DS 融合过程 | ✅ 完整访问 | ❌ 不可见 | 仅输出置信度 |
| 诊断结论 | ✅ 生成 | ✅ 只读 | 自然语言输出 |
| 健康分数 | ✅ 计算 | ✅ 只读 | 0-100 数值 |
| 维护建议 | ✅ 生成 | ✅ 只读 | 运维可执行 |
| 设备基础信息 | ✅ 管理 | ✅ 查询 | 设备树、测点 |
| 告警事件 | ✅ 生成 | ✅ 处置 | 双向同步 |

## 七、技术决策记录

### 7.1 API 网关选型

| 选项 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| Kong | 成熟、插件丰富 | 部署复杂 | ⭐ |
| Traefik | 轻量、易配置 | 插件较少 | |
| 直接 Express | 简单、无额外依赖 | 功能有限 | 初期可用 |

**决策**: 初期使用 Express 中间件，后期迁移 Kong

### 7.2 前端共享策略

| 选项 | 说明 | 推荐 |
|------|------|------|
| 独立代码库 | 两平台完全独立 | |
| 共享组件库 | 提取 `@xilian/ui` | ⭐ |
| 微前端 | Module Federation | 过度复杂 |

**决策**: 提取共享 UI 组件库，两平台独立构建

### 7.3 数据同步策略

| 数据类型 | 同步方式 | 频率 |
|---------|---------|------|
| 诊断结论 | API 调用 + Redis 缓存 | 实时 |
| 设备信息 | 数据库只读副本 | 准实时 |
| 历史数据 | ClickHouse 查询 | 按需 |
| 告警事件 | Kafka 消费 | 实时 |

## 八、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| API 延迟增加 | 应用平台响应变慢 | Redis 缓存、CDN |
| 数据不一致 | 用户看到过期数据 | 版本号 + 强制刷新 |
| 部署复杂度 | 运维成本增加 | Kubernetes + Helm |
| 团队学习曲线 | 开发效率下降 | 培训 + 文档 |
